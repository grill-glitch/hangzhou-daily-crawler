## 问题分析

查看 `/root/metrolist6.log` 发现网易云搜索接口返回 404 错误：
```
Response body: {"code":404,"message":"接口未找到！"}
```

对比 `api-enhanced` 仓库的实现后，发现两个关键错误：

1. **e_r 参数类型错误**：使用了布尔值 `false`，应该是字符串 `"false"`
2. **签名路径错误**：加密和签名时使用了转换后的路径 `/eapi/...`，应该使用原始路径 `/api/...`

## 解决方案

参考 api-enhanced 的实现原理：

### api-enhanced 正确流程：
```javascript
// 签名使用原始 uri
const message = "nobody/api/cloudsearch/pcuse{json}md5forencrypt"
// 加密数据使用原始 uri
const eapiData = "/api/cloudsearch/pc-36cd479b6b5-{json}-36cd479b6b5-{md5}"
// 最终请求 URL 使用转换后的路径
url = API_DOMAIN + '/eapi/cloudsearch/pc'
```

### 修改内容

文件：`Metrolist/netease/src/main/kotlin/com/metrolist/netease/NeteaseCloudMusicLyricsProvider.kt`

1. 将 `e_r` 参数从 `false` 改为 `"false"`（两处：搜索和歌词获取）
2. 修复 `eapiRequest` 中的签名和加密数据：
   - 签名消息：`"nobody${path}use..."`（使用原始 `path`）
   - 加密数据：`"$path-$EAPI_MAGIC-..."`（使用原始 `path`）
   - 注释说明：只有最终请求 URL 使用 `transformedPath`

## 新发现的问题（2026-03-26 19:45）

搜索功能已恢复（返回 200），但歌词获取失败：`LyricsHelper: NeteaseCloudMusic failed: No body in response`

**根本原因：** 歌词 API (`/api/song/lyric/v1`) 的响应结构是直接的 JSON 对象，不包含 `body` 嵌套字段。但原代码错误地尝试访问 `jsonObj.body`，导致异常。

### 修复步骤：

1. **移除 `body` 字段访问**，直接使用顶层 `jsonObj`
2. **移除 `e_r` 参数**，保持与 api-enhanced 一致（默认明文响应）
3. **解析 `lrc` 和 `tlyric` 字段**：
   - `lrc` 是对象，包含 `lyric` 字段（歌词文本）
   - `tlyric` 是顶层可选的翻译歌词字段

### 代码变更：

- 修改 `fetchLyrics`：直接解析 `lrc` 而非 `body.lrc`
- 删除 `e_r` 参数（歌词请求）
- 支持从 `code` 或 `status` 字段获取状态码

## 二次问题（2026-03-26 20:15）

在首次修复后，出现新的异常：
```
Element class kotlinx.serialization.json.JsonObject (Kotlin reflection is not available) is not a JsonPrimitive
```

**根本原因：** 某些歌曲（如 "Pretty Girl"）的响应中，`lrc.lyric` 字段不是字符串（`JsonPrimitive`），而是 `JsonObject`。原代码使用 `.jsonPrimitive` 直接转型导致异常。

### 二次修复：

- 对 `lrc.get("lyric")` 进行类型检查
- 若为 `JsonPrimitive` → 取 `.content`
- 若为 `JsonObject` → 转为字符串（`.toString()`），得到预期格式如 `{"t":0,"c":[...]}`
- 同样处理 `tlyric` 字段，增加鲁棒性

## 三次问题（2026-03-26 20:40）

网易云歌词返回的 `lrc.lyric` 是 JSON 字符串（可能多层嵌套），Metrolist 无法直接显示为可读歌词，需要转换为 LRC 格式。

**添加转换逻辑：**
- 新增 `convertNeteaseJsonLyric()` 函数，解析 `{"t":..., "c":[...]}` 结构
- 提取 `t`（时间，毫秒）转换为 `[MM:SS.xx]` 时间标签
- 提取 `c` 数组中的 `tx` 字段并拼接为歌词行
- 支持 lyric 中包含多行 JSON 对象（以 `\n` 分隔）
- 同时处理 `tlyric`（翻译歌词）字段，应用相同转换

现在歌词将以标准 LRC 格式返回，可以被 `parseLyrics()` 正确解析并显示逐行高亮。

## 四次问题（2026-03-26 21:10）

**问题发现：** 日志显示即使添加转换函数，返回的歌词仍是原始 JSON 字符串（preview: `{"t":0,"c":[...]}`），显示会异常。

**根本原因：** `lrc.lyric` 字段在 JSON 响应中是**字符串类型**（`JsonPrimitive`），不是 `JsonObject`。之前代码只针对 `JsonObject` 调用转换，`JsonPrimitive` 直接返回 `.content`，导致转换未执行。

### 最终修复：

- 统一从 `lyric` 提取原始字符串：`JsonPrimitive` → `content`，`JsonObject` → `toString()`
- 对提取出的**任何**原始字符串，都调用 `convertNeteaseJsonLyric()` 进行转换
- 同样处理 `tlyric`

这样无论 `lyric` 在 API 响应中是对象还是对象字符串，最终输出的都是标准 LRC 格式（以 `[` 开头），可以被 Metrolist 正常解析。
